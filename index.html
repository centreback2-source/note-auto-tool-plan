note 自動投稿ツール — 全体設計図

Architecture Overview / 開発者向けレビュー資料

Python + CustomTkinter Playwright Claude API (Anthropic) DALL-E 3 (OpenAI) Windows EXE 配布

このツールでできること

🎙

体験をヒアリング

AIがユーザーの日常・経験・悩みを対話形式で引き出し、「売れるジャンル」と読者像を自動で見つけ出す。

✍️

記事を全自動執筆

タイトル案10本の採点・本文（有料10,000字以上・無料3,000〜5,000字）・アイキャッチ画像を一気通貫で生成する。

📤

note へ自動投稿

ブラウザを自動操作してタイトル・本文・アイキャッチ・価格を設定し、下書き保存または公開まで実行する。

各フェーズの詳細

Phase A ─ ヒアリング対話　（Claude Haiku 4.5）
  • 起動セリフは固定（変更不可）。AIキャラクターは「横から共有する仲間」のスタンス
  • 1回1問ルール厳守。複数質問は禁止
  • 相手の言葉を「オウム返し＋意味の言語化」してから次へ進む
  • 直接聞かず体験を掘る迂回質問で6項目を引き出す
  • ①発信ジャンル ②ターゲット読者 ③深い悩み ④最悪の未来 ⑤解決策 ⑥最高の未来 が揃ったら自動で Phase B へ
  • 会話ログ全文を hearing.json に保存
Phase B ─ ジャンル・ペルソナ確定　（Claude Sonnet 4.6）
  • 会話ログから100ジャンルリストへ自動マッピング → 最適ジャンルを1つ自動確定
  • ペルソナ8項目（名前・年齢・職業・悩み・購入理由 等）を1回のAPIコールで出力
  • ジャンルを10区画に分割（カバレッジマップ生成）
  • 有料記事1本 + 無料記事9本の割り当てを自動決定
  • ユーザーの承認なしにそのまま Phase C へ移行（確認不要・自動進行）
  • 結果は画面に表示する
Phase C ─ 10記事の連続生成　（全自動）
  • 各記事について 8ステップを順番に実行（右欄参照）
  • 有料記事: 10,000字以上。無料記事: 3,000〜5,000字
  • 品質チェックで NG なら最大2回まで自動リトライ
  • 1本生成完了ごとに coverage_map.json を更新
  • 10本完成で完了通知
  • 途中でストップしても、次回起動時に続きから再開できる

Phase C ─ 1記事あたりの8ステップ

1

タイトル10案生成 + 採点 + 自動採用

30文字前後のタイトルを10案生成。ターゲット明確性・クリック強度・内容一致性など5軸で100点満点採点。最高得点を自動採用。

Claude Sonnet 4.6
2

リサーチ（web検索で素材収集）

web_search ツールを有効化し、現状・問題点・失敗事例・成功事例・海外研究など8項目の素材をURL付きで収集する。

Claude Sonnet 4.6 web_search 有効
3

結起承転 構成生成

有料記事: 無料ゾーン（結→起→承→転の渇望を作る）＋「ここから有料」マーカー＋有料ゾーン（具体手順）。無料記事: 結起承転で完結。

Claude Sonnet 4.6
4

本文生成

冒頭は読者の心の声を「」で3〜4つ。本文はです・ます調統一、同語尾3連続禁止、感情6:理解4の比率、情景描写を含める。AI語禁止。

Claude Sonnet 4.6
5

品質チェック

構成チェック・文体チェック・読者チェック・文字数チェックの4カテゴリ全項目を確認。NGがあれば最大2回まで自動リトライ。

Claude Haiku 4.5
6

アイキャッチ画像生成

8ステップ自動判断（タイトル構造分類→人物使用判断→ペルソナ性別年齢参照→色設計）でDALL-E 3用英語プロンプトを自動生成・画像保存。

DALL-E 3 (OpenAI)
7

note へ投稿（ブラウザ自動操作）

Playwrightでブラウザを操作し、タイトル・本文・タグ・価格を入力。アイキャッチをアップロード。有料記事は境界線を自動設定。デフォルトは下書き保存。

Playwright
8

進捗を記録

coverage_map.json の該当区画を used: true、タイトル・生成日時を書き込み。次の記事へ進む。10本完了で完了通知を表示。

ローカル保存

AI 構成とコスト概算

工程 プロバイダ モデル 選択理由
ヒアリング対話・ジャンルマッピング Anthropic Claude Haiku 4.5 会話制御など軽量タスク。低コスト・高速
タイトル採点・構成生成 Anthropic Claude Sonnet 4.6 採点ロジックや構造設計に高精度が必要
リサーチ（web_search付き） Anthropic Claude Sonnet 4.6 Anthropic 組込の web_search ツールを使用
本文生成（10,000字以上） Anthropic Claude Sonnet 4.6 日本語長文・文体ルール遵守に最高品質が必要
本文品質チェック Anthropic Claude Haiku 4.5 チェックリスト評価は軽量で十分
アイキャッチ画像生成 OpenAI DALL-E 3 Standard ナレッジ6章のプロンプト設計がDALL-E 3前提
コスト目安: 10記事1セットの生成で 300〜500円程度（ユーザー負担）。ナレッジドキュメントをくり返し参照する部分は Prompt Caching（5分TTL） を使いコストを削減。APIキーはユーザーが自分で用意・保持し、運営側には渡らない。

🔐 OS Keychain（Credential Manager）

以下の情報はOSのKeychain（Windows Credential Manager / macOS Keychain）に保存します。テキストファイルには書き出しません。


  • Anthropic API キー
  • OpenAI API キー
  • note.com メールアドレス
  • note.com パスワード

なお session.json（ログイン済みブラウザ情報）はローカルファイルとして保持します。

セキュリティ設計

🔑

APIキー・認証情報はOS Keychain保存

Python keyring ライブラリ経由でWindowsCredential Manager / macOS Keychain に保存。設定ファイルや環境変数には書き出さない。

📡

外部送信先は3か所のみ

Anthropic（Claude API）/ OpenAI（DALL-E 3）/ note.com の3か所のみ。運営サーバーへの送信はゼロ。ユーザーデータはすべてユーザーのPCとAPIプロバイダー間で完結する。

🫣

ログのマスキング

アプリログ（app.log）へ書き出す際に、APIキー・パスワードなどの機密情報を自動でマスク処理する。

📂

オープンソース化（推奨）

GitHubでソースコードを公開することで「何がどこに送信されるか」をユーザーが自分で確認できる透明性を担保する。

モバイル（iOS/Android）への進化の布石

📱

今回はPC版のみ

Playwright はモバイル非対応のため、PC版 EXE の完成を最優先。モバイル対応は Phase 2 として明示的に後回し。

🔌

執筆コアと投稿を分離

Phase A〜C（ヒアリング・生成）は note.com 非依存の設計にする。note_client/ のみが Playwright 依存。

🔄

将来のAPI化に備えた設計

各 phase を CLI/API として呼び出せる関数シグネチャで設計。モバイル向けには執筆コアをWebAPI化、投稿は手動コピペ or note公式APIに切替の選択肢を残す。

実装着手前に決めたい事項

📄

ナレッジドキュメントの同梱方法

EXEにナレッジ全文を同梱するか、プロンプトに「焼き込む」か。同梱するとEXEを解凍した際にナレッジ全文が見えてしまう。焼き込むと更新が手間になる。

🔍

note 有料境界線 UI の事前検証

note の編集画面で「ここから有料」ボタンのPlaywright操作が可能かどうかを、実際のブラウザで手動確認が必要。note の UI 変更頻度が高いため要事前検証。

💡

ヒアリングのモデル選択（ハイブリッド戦略の是非）

会話全体を低コストの Claude Haiku で進め、Phase 移行の判定だけ Claude Sonnet で行うハイブリッド戦略の検討。コスト削減効果はあるが、実装の複雑度が上がる。
